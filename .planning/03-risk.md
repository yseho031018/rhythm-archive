# 하루톡 위험 관리

> 하루톡 개발과 최종 평가에서 발생할 수 있는 위험과 대응 방안을 정리한다.

## 위험 목록

| ID | 카테고리 | 위험 내용 | 영향도 | 가능성 | 대응 방안 |
|---|---|---|---|---|---|
| R1 | 범위 | 질문형 기록, 조회, 감정잔디, 통계, AI 연결을 모두 고도화하면 개인 프로젝트 범위를 초과할 수 있다. | 높음 | 높음 | Must 기능을 먼저 완료하고 실제 AI, 알림, 필터는 향후 확장으로 관리한다. |
| R2 | 생성 신뢰도 | 자동 생성된 한 줄이 사용자가 선택하지 않은 사실을 포함할 수 있다. | 높음 | 중간 | 입력값을 기분·키워드·만족도로 제한하고 다시 생성·직접 수정 기능을 제공한다. |
| R3 | 분석 신뢰도 | 기록 수가 적은 상태에서 활동과 감정의 관계를 과하게 해석할 수 있다. | 높음 | 높음 | 최소 표본과 평균 차이 기준을 두고 단정이 아닌 관찰 문장으로 표현한다. |
| R4 | 저장 | 저장 실패나 데이터 형식 변경으로 기록을 불러오지 못할 수 있다. | 높음 | 중간 | Repository로 저장 책임을 분리하고 실패 테스트와 사용자 오류 안내를 추가한다. |
| R5 | 성능 | 통계 집계와 큰 이미지가 저사양 기기에서 화면 반응을 느리게 만들 수 있다. | 중간 | 낮음 | 단순 로컬 집계, 모바일 최대 폭, 상태별 이미지 에셋을 사용한다. |
| R6 | AI 의존 | AI Agent가 생성한 코드의 역할을 이해하지 못하면 발표 Q&A에서 설명하기 어렵다. | 높음 | 중간 | 직접 실행·수정·테스트하고 설명 가능한 결과만 반영한다. |
| R7 | 배포 | GitHub Pages 설정이나 base href 문제로 앱 URL이 열리지 않을 수 있다. | 높음 | 중간 | GitHub Actions 자동 배포와 로컬 서버·MP4 백업을 함께 준비한다. |
| R8 | 발표 | 구현 내용은 충분하지만 슬라이드를 읽는 발표가 될 수 있다. | 중간 | 중간 | 시각 중심 슬라이드와 별도 말하기 카드를 사용해 반복 연습한다. |

## 우선 대응 대상

가장 먼저 관리하는 위험은 **범위 초과(R1)**, **생성 신뢰도(R2)**, **배포 실패(R7)** 다.

하루톡의 1차 완료 기준은 세 번의 선택으로 한 줄을 만들고, 사용자가 결과를 제어하며, 저장된 기록을 감정잔디와 생활 패턴에서 다시 확인하는 것이다. 실제 AI와 복잡한 데이터베이스보다 이 사용자 흐름의 안정성을 우선한다.

## 위험 대응 체크리스트

- [x] Must 기능과 향후 확장 기능을 구분했다.
- [x] 자동 생성 결과의 다시 생성·직접 수정을 제공한다.
- [x] 패턴 분석에 최소 표본 기준을 적용한다.
- [x] 저장 실패와 핵심 사용자 흐름을 테스트한다.
- [x] AI Agent 결과를 직접 실행·검증한다.
- [x] Flutter 구조, 빌드, 배포, 테스트 문서를 작성했다.
- [ ] 공개 URL, 로컬 서버, 30초 MP4 백업을 최종 확인한다.
