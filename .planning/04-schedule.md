## 2. 주차별 상세 계획

### 10주차 — 기획 및 범위 확정

- 프로젝트 비전과 문제 정의를 정리한다.
- 핵심 사용자 시나리오 3개를 확정한다.
- 기능을 Must/Should/Could/Won't로 분류한다.
- WBS, 일정, 위험 관리 문서를 작성한다.

**완료 기준:** 교수님이 요구한 기획 문서 세트를 저장소에서 확인할 수 있다.

### 11주차 — 설계 결정 및 발표 준비 기반 마련

- Flutter 선택 이유를 ADR-0001로 정리한다.
- Layered Architecture 선택 이유를 ADR-0002로 정리한다.
- Isar 기반 로컬 우선 저장 선택 이유를 ADR-0003으로 정리한다.
- WBS/Gantt 페이지를 GitHub Pages에서 열 수 있게 정리한다.
- 중간 발표자료 초안을 작성한다.

**완료 기준:** 플랫폼, 아키텍처, 데이터 저장 방식을 말로 설명할 수 있고 WBS/Gantt URL을 열 수 있다.

### 12주차 — 중간 발표 및 Flutter Web 데모 선행 구현

- 중간 발표자료를 `presentation.html`로 변환한다.
- Flutter 프로젝트를 생성하고 Web 데모를 구현한다.
- 오늘 입력, Particle Canvas, 히스토리, 패턴 카드 흐름을 먼저 시연 가능하게 만든다.
- GitHub Pages 루트에서 앱 데모가 바로 열리도록 배포한다.
- `docs/setup.md`, `docs/architecture.md`를 작성한다.

**완료 기준:** 아래 URL 3개를 발표 중 바로 열 수 있다.

```text
https://yseho031018.github.io/rhythm-archive/
https://yseho031018.github.io/rhythm-archive/presentation.html
https://yseho031018.github.io/rhythm-archive/wbs-gantt.html
```

### 13주차 — 아키텍처 분리 및 로컬 DB 연동

- 현재 단일 파일 중심 데모를 Presentation/Application/Domain/Data 레이어로 분리한다.
- Riverpod을 도입해 입력 상태와 기록 목록 상태를 분리한다.
- `RhythmEntry` Entity와 Repository Interface를 Domain Layer로 이동한다.
- Isar Collection 모델과 Repository 구현체를 작성한다.
- 임시 저장 방식인 `shared_preferences`를 Isar 로컬 DB로 교체한다.

**완료 기준:** 저장과 조회가 Isar 기반으로 동작하고, 각 파일이 어느 레이어에 속하는지 설명할 수 있다.

### 14주차 — 기능 고도화 및 테스트 시작

- 히스토리에서 날짜별 조회, 수정, 삭제 흐름을 보강한다.
- 패턴 화면에서 에너지 평균, 감정 빈도, 활동 빈도 힌트를 개선한다.
- 시간이 가능하면 데이터 내보내기와 리마인드 알림 중 하나를 선택해 구현한다.
- 위젯 테스트와 수동 검증 기록을 보강한다.

**완료 기준:** 핵심 Must 기능이 안정적으로 동작하고, 테스트 또는 수동 검증 근거가 남아 있다.

### 15주차 — 최종 발표 준비

- README, setup, architecture 문서를 최종 상태에 맞게 수정한다.
- 최종 발표자료와 Q&A 대비 문서를 작성한다.
- 앱 데모 URL과 발표자료 URL을 다시 점검한다.
- 발표 시간에 맞춰 데모 흐름을 리허설한다.

**완료 기준:** 앱 데모, 기술 구조, AI Agent 활용 방식, 남은 한계를 모두 설명할 수 있다.

## 3. 최소 완성 범위

최종 제출 시 반드시 완성해야 하는 최소 범위는 다음과 같다.

| 구분 | 최소 완성 기준 |
|------|----------------|
| 기록 | 에너지, 감정, 활동, 메모를 입력하고 저장할 수 있다. |
| 조회 | 저장된 기록을 히스토리에서 다시 볼 수 있다. |
| 시각화 | 입력한 기록이 Particle Canvas에 기본 형태로 반영된다. |
| 분석 | 1주 단위의 간단한 통계 힌트를 제공한다. |
| 오프라인 | 로컬 저장 기반으로 네트워크 없이 핵심 흐름이 동작한다. |
| 문서 | 기획, WBS, 일정, 위험, ADR, README, setup, architecture를 갖춘다. |

## 4. 일정 변경 사유

초기 계획에서는 12주차에 도메인/데이터 구조를 먼저 만들고 이후 화면을 붙이는 흐름을 생각했다.  
하지만 중간 발표에서 **즉시 열 수 있는 발표 URL과 데모 URL**이 필요해졌기 때문에, Flutter Web 데모와 GitHub Pages 배포를 먼저 완료했다.

이 때문에 다음 작업은 순서가 바뀌었다.

- 원래 먼저 하려던 작업: Domain, Repository, Isar DB
- 실제 먼저 한 작업: Flutter Web 데모, 발표자료 HTML, GitHub Pages 배포
- 이후 보강할 작업: Riverpod, Isar, 레이어 분리, 테스트
