## 2. 주차별 상세 계획

### 10주차 — 기획 및 범위 확정

- 프로젝트 비전과 문제 정의를 정리한다.
- 핵심 사용자 시나리오 3개를 확정한다.
- 기능을 Must/Should/Could/Won't로 분류한다.
- WBS와 위험 관리 문서를 작성한다.

**완료 기준:** 교수님이 요구한 기획 문서 세트를 저장소에서 확인할 수 있다.

### 11주차 — 설계 및 개발 환경 구축

- Flutter 프로젝트를 생성한다.
- Clean Architecture 기준의 폴더 구조를 만든다.
- Riverpod, Isar 등 필수 패키지를 추가한다.
- Flutter 선택 이유를 ADR로 기록한다.

**완료 기준:** 개발 환경에서 기본 앱이 실행되고, 주요 기술 선택 이유를 설명할 수 있다.

### 12주차 — 핵심 데이터 구조와 중간 발표 준비

- 일일 리듬 기록 Entity를 정의한다.
- Repository 인터페이스와 UseCase 초안을 작성한다.
- Isar Collection 모델을 설계한다.
- 중간 발표용 데모 흐름을 준비한다.

**완료 기준:** 입력 데이터가 어떤 구조로 저장될지 설명할 수 있다.

### 13주차 — 일일 기록 기능 구현

- 에너지 레벨, 감정 키워드, 활동 태그 입력 UI를 구현한다.
- 입력한 데이터를 로컬 DB에 저장한다.
- 저장된 데이터를 다시 불러온다.
- 입력값을 Particle Canvas에 기본 형태로 반영한다.

**완료 기준:** 사용자가 하루 기록을 남기고 화면에서 결과를 확인할 수 있다.

### 14주차 — 히스토리와 패턴 확인 기능 구현

- 날짜별 기록 조회 화면을 구현한다.
- 주간 에너지 평균과 감정 빈도를 계산한다.
- 간단한 패턴 힌트를 카드 형태로 보여준다.
- Should 기능 중 가능한 항목을 선택적으로 확장한다.

**완료 기준:** 여러 날의 기록을 기반으로 기본 통계와 힌트를 확인할 수 있다.

### 15주차 — 테스트, 문서화, 최종 발표

- 핵심 흐름을 수동 검증하고 결과를 문서화한다.
- 가능한 범위에서 단위 테스트 또는 위젯 테스트를 작성한다.
- README에 설치, 실행, 빌드 방법을 정리한다.
- 최종 발표 자료와 예상 질문 답변을 준비한다.

**완료 기준:** 앱 데모와 기술 설명을 발표 시간 안에 안정적으로 진행할 수 있다.

## 3. 최소 완성 범위

최종 제출 시 반드시 완성해야 하는 최소 범위는 다음과 같다.

| 구분 | 최소 완성 기준 |
|------|----------------|
| 기록 | 에너지, 감정, 활동을 입력하고 저장할 수 있다. |
| 조회 | 저장된 기록을 날짜별로 다시 볼 수 있다. |
| 시각화 | 입력한 기록이 Particle Canvas에 기본 형태로 반영된다. |
| 분석 | 1주 단위의 간단한 통계 힌트를 제공한다. |
| 문서 | 기획, WBS, 일정, 위험, ADR, README를 갖춘다. |
