# ADR-0003 — Isar 기반 로컬 우선 데이터 저장 선택

**상태**: Accepted  
**작성일**: 2026-05-26  
**작성자**: 윤세호

## 배경

Rhythm은 사용자가 매일의 감정, 에너지, 활동을 짧게 기록하고 이를 Wave Graph Canvas로 시각화하는 개인용 다이어리 앱이다.  
핵심 시나리오에는 네트워크가 없는 상황에서도 기록을 남기고, 이전 기록을 다시 조회하며, 1주 이상 누적된 데이터를 기반으로 패턴을 확인하는 흐름이 포함되어 있다.

따라서 초기 버전에서는 서버나 계정 기능보다 **오프라인에서도 안정적으로 동작하는 로컬 저장 구조**가 더 중요하다.  
또한 7주 1인 프로젝트 범위 안에서 개발해야 하므로, 백엔드 구축과 인증, 동기화까지 포함하면 프로젝트 범위가 과도하게 커질 수 있다.

## 고려한 대안

| 대안 | 장점 | 단점 / 제외 이유 | 적합도 |
|------|------|------------------|--------|
| SharedPreferences | 설정값처럼 간단한 데이터를 저장하기 쉽다. | 일일 기록, 감정 목록, 활동 태그처럼 구조화된 데이터를 관리하기 어렵다. | ★★☆☆☆ |
| SQLite / Drift | 안정적이고 범용적인 관계형 DB 구조를 사용할 수 있다. | SQL/테이블 설계 부담이 있고, 초기 개발 속도가 느려질 수 있다. | ★★★☆☆ |
| Firebase Firestore | 클라우드 동기화와 백업이 쉽다. | 계정, 네트워크, 보안 규칙, 무료 사용량 관리까지 고려해야 해서 범위가 커진다. | ★★★☆☆ |
| Supabase | PostgreSQL 기반 백엔드와 인증을 함께 사용할 수 있다. | 서버 중심 구조가 되어 초기 목표인 오프라인 우선 앱과 거리가 생긴다. | ★★☆☆☆ |
| Isar | Flutter/Dart와 잘 맞고, 구조화된 로컬 데이터를 빠르게 저장할 수 있다. | Isar 사용법과 스키마 관리에 대한 학습이 필요하다. | ★★★★★ |

## 결정

Rhythm의 초기 데이터 저장 방식으로 **Isar 기반 로컬 우선(Local-first) 구조**를 선택한다.

- 백엔드 서버는 v0.1.0 범위에서 제외한다.
- 일일 리듬 기록, 감정 키워드, 활동 태그, 관찰 노트는 로컬 DB에 저장한다.
- Repository Interface는 Domain Layer에 두고, Isar 구현체는 Data Layer에 둔다.
- 추후 백업이나 클라우드 동기화가 필요해지면 Repository 구현체를 확장한다.

## 선택 이유

- **오프라인 완전 지원**  
  Rhythm은 매일 짧게 기록하는 앱이므로, 네트워크 상태와 상관없이 입력, 저장, 조회가 가능해야 한다.

- **개인 프로젝트 범위에 적합**  
  서버, 인증, 동기화를 제외하면 핵심 기능인 기록, 시각화, 히스토리 구현에 시간을 집중할 수 있다.

- **구조화된 기록 데이터 관리**  
  에너지 레벨, 감정 키워드, 활동 태그, 메모, 날짜 정보를 하나의 기록 모델로 저장하고 조회하기 좋다.

- **Layered Architecture와의 연결**  
  Domain Layer는 Repository Interface만 알고, Data Layer가 Isar 구현을 담당하면 나중에 저장 방식을 바꿔도 핵심 로직의 변경을 줄일 수 있다.

- **발표 Q&A 대비**  
  “왜 백엔드를 쓰지 않았는가?”라는 질문에 대해, 초기 버전의 목적이 개인 기록과 오프라인 동작이기 때문이라고 설명할 수 있다.

## 결과 (예상되는 영향)

### 긍정

- 네트워크 없이도 핵심 기능을 사용할 수 있다.
- 백엔드 구축 시간을 줄이고 앱 기능 구현에 집중할 수 있다.
- 사용자 기록이 기기 안에 저장되어 개인 다이어리 앱의 성격과 잘 맞는다.
- Repository 구조를 통해 추후 Firebase, Supabase, 백업 기능으로 확장할 여지를 남긴다.

### 부정 / 제약

- 여러 기기 간 자동 동기화는 초기 버전에서 지원하지 않는다.
- 기기를 변경하거나 앱을 삭제하면 데이터 보존 문제가 생길 수 있다.
- Isar 스키마 변경 시 마이그레이션을 신경 써야 한다.

## 위험 및 대응

- **데이터 유실 위험**  
  → v0.1.0에서는 로컬 저장을 우선하고, Should 기능으로 JSON/CSV 내보내기를 준비한다.

- **스키마 변경 위험**  
  → 초기 모델을 단순하게 유지하고, 변경 사항은 ADR 또는 개발 메모에 남긴다.

- **AI 의존 위험**  
  → AI가 생성한 Isar 모델과 Repository 코드는 직접 읽고, 각 필드와 저장 흐름을 발표에서 설명할 수 있을 때만 반영한다.

## 후속 작업

- [ ] `RhythmEntry` Entity 정의
- [ ] `RhythmRepository` Interface 작성
- [ ] Isar Collection 모델 작성
- [ ] 저장, 조회, 수정, 삭제 기본 CRUD 구현
- [ ] 샘플 데이터로 히스토리 화면과 통계 카드 연결

## 발표 시 설명 포인트

- Rhythm은 개인 감정 기록 앱이므로 초기 버전에서는 서버보다 로컬 저장이 더 적합하다.
- 백엔드, 인증, 동기화를 제외해 7주 개인 프로젝트 범위 안에서 완성 가능성을 높였다.
- Isar는 Flutter와 잘 맞고 구조화된 로컬 데이터를 관리하기 좋아 선택했다.
- Repository Interface를 Domain Layer에 두어 나중에 클라우드 백업으로 확장할 수 있게 했다.
