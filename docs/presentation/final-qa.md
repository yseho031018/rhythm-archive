# 하루톡 최종 Q&A 답변집

## 1. 왜 Flutter를 선택했나요?

모바일 중심 UI와 토리 캐릭터 화면을 빠르게 반복 개발하면서, 발표에서는 Web Release를 GitHub Pages URL로 바로 보여줘야 했기 때문입니다. 하나의 Dart 코드베이스로 앱, 테스트, Web 시연을 관리할 수 있다는 점이 가장 컸습니다. ADR-0001에 기록했습니다.

## 2. 앱 구조는 어떻게 되어 있나요?

화면, DiaryController, 핵심 모델과 패턴 분석, Repository 저장소로 나눴습니다. 화면은 입력과 표시, Controller는 기록 흐름, PatternAnalysis는 생활 패턴 계산, Repository는 Drift/SQLite 저장을 담당합니다.

## 3. 왜 완전한 Clean Architecture나 Riverpod을 쓰지 않았나요?

개인 프로젝트의 남은 기간과 기능 규모를 고려하면 보일러플레이트가 커질 수 있었습니다. 대신 핵심 책임을 분리하고 테스트 가능한 Repository Pattern을 적용했습니다. 계획과 실제 구현의 차이는 ADR-0002에 기록했습니다.

## 4. 왜 Drift/SQLite를 사용했나요?

오프라인 기록을 구조화된 스키마와 트랜잭션으로 관리하면서 Flutter Web 시연도 유지해야 했기 때문입니다. Drift는 네이티브 SQLite와 WebAssembly SQLite를 같은 Dart API로 사용할 수 있습니다. 기존 SharedPreferences 기록은 최초 실행에 자동 이전하며, 이 결정은 ADR-0003에 기록했습니다.

## 5. AI는 실제 모델인가요?

현재 한 줄 생성은 규칙 기반입니다. 외부 AI가 없어도 발표와 오프라인 기록 흐름이 안정적으로 동작하도록 선택했습니다. 실제 AI 연결은 향후 확장이고, 연결하더라도 사용자가 선택한 사실만 입력하며 실패 시 규칙 기반 생성으로 대체할 계획입니다.

## 6. 생활 패턴은 어떻게 계산하나요?

누적 기록을 키워드별로 묶고 평균 만족도, 전체 평균과의 차이, 가장 자주 함께 나타난 기분을 계산합니다. 표본이 두 번 미만이거나 만족도 차이가 작으면 관계를 단정하지 않습니다.

## 7. 개발 환경은 어떻게 구성했나요?

Windows에서 Flutter 3.44.0 stable과 Dart 3.12.0을 사용했습니다. Chrome Web을 기본 실행 환경으로 사용하고 Git과 GitHub CLI로 버전 관리와 저장소 작업을 합니다. 설치와 확인 명령은 docs/setup.md에 정리했습니다.

## 8. 빌드와 배포의 차이는 무엇인가요?

빌드는 Flutter 코드를 브라우저용 HTML, JavaScript, 에셋으로 변환하는 과정입니다. 배포는 그 결과를 GitHub Pages에 올려 다른 PC에서도 URL로 실행 가능하게 만드는 과정입니다.

## 9. 어떤 테스트를 했나요?

Controller의 생성·저장·수정·삭제, 사용자 키워드, 과거 날짜, 실패 처리와 백업 미리보기·유효성 검사를 단위 테스트했습니다. Drift 저장소의 실제 쿼리, 기존 기록 이전, 백업 복원 트랜잭션과 패턴 분석 규칙도 검증했습니다. 기록 생성부터 저장까지, 이전 질문과 선택 유지, 월간·연간 감정잔디, 빈 상태 행동은 통합 위젯 테스트로 확인했습니다. 현재 50개가 통과합니다.

## 10. 성능 최적화는 무엇을 했나요?

외부 서버 호출 없이 핵심 흐름을 로컬에서 처리했고, 화면 최대 폭을 560px로 제한했습니다. 연간 감정잔디는 필요한 날짜 셀만 그리는 단순한 그리드로 구성했고, 패턴 분석은 작은 기록 목록에 대한 집계로 구현했습니다. 토리 이미지도 상태별 파일로 분리해 필요한 에셋만 사용합니다.

## 11. 가장 큰 시행착오는 무엇인가요?

자동 생성된 한 줄이 사용자가 선택하지 않은 사실을 포함하면 신뢰하기 어렵다는 점이 가장 큰 문제였습니다. 그래서 생성 입력을 기분, 키워드, 만족도로 제한하고, 다시 생성과 직접 수정 기능을 추가했습니다. 생활 패턴도 표본이 부족하면 관계를 단정하지 않도록 기준을 뒀습니다.

## 12. AI Agent는 어디까지 사용했나요?

아이디어 검토, 문서 초안, Flutter 구현, 테스트 작성, 이미지 분리, 최종 평가 감사에 활용했습니다. 방향 결정과 범위 조절은 제가 했고, 생성 결과는 직접 실행·검토·수정한 뒤 반영했습니다. 프로젝트 작업 규칙은 AGENTS.md에 통합했습니다.

## 13. 앞으로 어떻게 활용할 수 있나요?

매일 30초 기록으로 사용자가 자신의 감정과 활동 관계를 돌아보는 데 활용할 수 있습니다. 이후 실제 로컬 AI 서버를 연결해 표현을 다양하게 하고, 여러 기기 동기화가 필요해지면 서버 저장을 추가할 수 있습니다.

## 14. 왜 로그인 기능이 없나요?

하루톡의 핵심은 로그인 절차 없이 바로 기록을 시작하고, 민감한 감정 기록을 사용자 기기에 보관하는 것입니다. 현재는 JSON 백업과 복원으로 사용자가 직접 기록을 옮기고 소유할 수 있게 했습니다. 여러 기기 자동 동기화가 필요해지는 시점에는 로그인과 서버 저장을 선택 기능으로 추가할 계획입니다.
