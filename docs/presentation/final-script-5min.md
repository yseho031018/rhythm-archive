### 2페이지 · 비전과 문제 정의 · 0:15–0:45

**[두 문제가 나타난 뒤 마지막 비전 문장을 강조한다.]**

하루톡의 비전은 긴 글을 쓰지 않아도 자신의 하루와 감정 흐름을 돌아볼 수 있게 하는 것입니다. 기존 다이어리는 매일 글을 길게 써야 한다는 부담 때문에 기록이 쉽게 끊기고, 기록이 쌓여도 나의 활동과 감정이 어떤 관계인지 알기 어렵다는 문제가 있습니다.

**→ 다음 페이지**

### 3페이지 · 사용자 시나리오 · 0:45–1:15

**[카드가 나타나는 순서대로 왼쪽에서 오른쪽으로 시선을 이동한다.]**

먼저 오늘의 기분을 고르고, 함께한 활동과 만족도를 차례로 선택합니다. 이 세 가지 선택을 마치면 앱이 사용자가 고른 사실만 사용해 오늘을 한 줄로 정리합니다. 잘못 선택했다면 이전 질문으로 돌아가도 답변이 유지되고, 결과 한 줄은 다시 생성하거나 직접 수정할 수 있습니다.

**→ 다음 페이지**

### 4페이지 · WBS와 진행 과정 · 1:15–1:40

**[게이지가 채워지는 동안 진행 단계를 왼쪽에서 오른쪽으로 짚는다.]**

프로젝트는 기획, 질문형 UX, 핵심 기록과 저장, 분석과 조회, 테스트와 발표 준비 순서로 WBS를 구성했습니다. 최종 검증과 발표 준비까지 포함한 전체 21개 작업을 완료해 WBS 진척률 100%를 달성했습니다.

**→ 다음 페이지**

### 5페이지 · 기술과 앱 구조 · 1:40–2:15

**[레이어가 나타나는 순서대로 화면 → 흐름 → 규칙 → 저장 책임을 설명한다.]**

기술은 하나의 코드베이스로 모바일 UI와 Web 데모를 만들 수 있는 Flutter를 선택했습니다. 구조는 화면, Controller, 핵심 모델과 분석 규칙, Repository 저장소로 책임을 나눴습니다. Repository 인터페이스 덕분에 테스트에서는 실제 저장소 대신 메모리 저장소를 사용할 수 있습니다.

**→ 다음 페이지**

### 6페이지 · 구현과 시행착오 · 2:15–2:50

구현 과정에서는 사용자가 실수해도 부담 없이 복구할 수 있게 만드는 것을 중요하게 생각했습니다. 그래서 이전 질문 이동과 선택 유지, 결과 직접 수정, 확인 팝업을 제공했습니다. 저장은 Repository 경계를 유지한 채 Drift와 SQLite로 구현했으며, 기존 SharedPreferences 기록도 자동 이전하도록 만들었습니다. 이러한 기술 결정은 ADR 세 개에 기록했습니다.

**→ 다음 페이지**

### 7페이지 · 품질과 배포 · 2:50–3:25

**[검증 결과를 말한 뒤 나타나는 배포 파이프라인을 왼쪽에서 오른쪽으로 설명한다.]**

현재 정적 분석은 이슈 없이 통과했고, 이전 질문과 선택 유지, 월간·연간 감정잔디, Drift 저장과 JSON 백업·복원까지 총 53개 테스트가 통과합니다. GitHub Actions는 push마다 분석, 테스트, Web Release 빌드를 검증하고, 성공한 결과를 GitHub Pages에 배포합니다.

**→ 다음 페이지**

짧은 기록이 쌓이면 먼저 감정의 흐름이 보입니다. 이를 활동과 비교하면 친구와 함께한 날의 만족도처럼 생활의 관계를 발견할 수 있습니다. 하루톡은 이 관계를 단정하지 않고, 다음 생활 선택을 돌아보는 참고 자료로 제공합니다.
