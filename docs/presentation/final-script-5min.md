## 대본

안녕하세요. 세 번의 선택으로 오늘을 남기는 감정 다이어리, 하루톡을 개발한 윤세호입니다.

하루톡의 비전은 긴 글을 쓰지 않아도 자신의 하루와 감정 흐름을 돌아볼 수 있게 하는 것입니다. 기존 다이어리는 매일 글을 길게 써야 한다는 부담 때문에 기록이 쉽게 끊기고, 기록이 쌓여도 나의 활동과 감정이 어떤 관계인지 알기 어렵다는 문제가 있습니다.

하루톡에서는 토리가 기분, 함께한 일, 만족도를 하나씩 묻습니다. 사용자가 선택을 마치면 앱이 선택한 사실만 사용해 오늘을 한 줄로 정리합니다. 저장된 기록은 한줄 목록과 감정잔디에 나타나고, 누적되면 친구를 만난 날의 만족도처럼 생활 패턴을 보여줍니다.

프로젝트는 기획, 핵심 기록, 저장과 조회, 패턴 분석, 테스트와 발표 준비 순서로 WBS를 구성했습니다. 기술은 하나의 코드베이스로 모바일 UI와 Web 데모를 만들 수 있는 Flutter를 선택했습니다. 상태 관리는 현재 규모에 맞는 ChangeNotifier, 오프라인 저장은 Drift와 SQLite를 사용했습니다.

구조는 화면, Controller, 핵심 모델과 분석 로직, Repository 저장소로 책임을 나눴습니다. 화면은 입력과 표시를 담당하고, DiaryController는 기록 생성과 저장 흐름을 관리합니다. pattern_analysis는 UI와 분리된 순수 로직이며, Repository 인터페이스 덕분에 테스트에서는 실제 저장소 대신 메모리 저장소를 사용할 수 있습니다.

구현 과정에서 가장 고민한 부분은 자동 생성 결과를 사용자가 믿고 사용할 수 있게 만드는 것이었습니다. 그래서 선택하지 않은 사실은 만들지 않고, 결과를 다시 생성하거나 직접 수정할 수 있게 했습니다. 저장은 처음에 SharedPreferences로 검증한 뒤 Repository 경계를 유지한 채 Drift/SQLite로 교체했고, 기존 기록도 최초 실행에 자동 이전하도록 만들었습니다.

성능은 복잡한 서버 호출 없이 로컬에서 동작하게 했고, 모바일 화면 최대 폭을 제한했으며, 이미지 에셋을 상태별로 분리했습니다. 코드 품질은 lint와 책임 분리로 관리했고, 패턴 분석은 표본이 부족하면 관계를 단정하지 않도록 만들었습니다.

현재 정적 분석은 이슈 없이 통과했고, 기록 생성·저장·수정·삭제, Drift 저장과 이전, JSON 백업 복원, 패턴 분석 단위 테스트, 실제 기록 흐름과 화면 이동 통합 테스트까지 총 39개가 통과합니다. 빌드는 Flutter 코드를 Web Release 파일로 만드는 과정이고, 배포는 그 결과를 GitHub Pages에 올려 URL로 공개하는 과정입니다. 설치, 테스트, 배포 과정은 README와 별도 문서로 정리했습니다.

현재는 로그인 없이 Drift에 기록을 저장하고 JSON 백업으로 사용자가 데이터를 직접 소유하게 했습니다. 앞으로는 규칙 기반 한 줄 생성기를 실제 로컬 AI 서버로 교체하고, 여러 기기 자동 동기화가 필요해지면 같은 Repository 구조 뒤에 서버 저장을 추가할 수 있습니다. 이번 프로젝트에서는 AI Agent를 기획, 구현, 테스트, 문서 감사에 사용했으며, 생성 결과는 제가 검토하고 설명 가능한 범위만 반영했습니다.

하루톡은 일기를 대신 길게 써주는 앱이 아니라, 기록을 시작하기 쉽게 만들고 나의 생활을 돌아보게 하는 앱입니다. 감사합니다.

## 30초 시연 순서

1. 기분 `행복` 선택
2. 키워드 `친구` 선택
3. 만족도 `5점` 선택 후 한 줄 생성
4. 결과 저장
5. 한줄 목록 또는 감정잔디 반영 확인
6. 통계의 생활 패턴 문장으로 마무리
