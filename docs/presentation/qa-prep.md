# Rhythm Q&A 단골 질문 대비

## 목적

중간 발표 후 자주 나올 수 있는 질문에 대비하기 위한 답변 정리 문서다.  
답변은 짧고 솔직하게 하되, 프로젝트의 선택 이유와 본인이 이해한 부분이 드러나도록 말한다.

## 2분 발표 대본

안녕하세요. 앱 프로그래밍 개인 프로젝트로 **Rhythm**이라는 앱을 준비한 윤세호입니다.

Rhythm은 **감정의 파도를 시각화하는 다이어리 앱**입니다. 기존 다이어리 앱은 매일 길게 기록해야 해서 부담이 있고, 기록이 쌓여도 리스트나 숫자로만 보여주는 경우가 많다고 생각했습니다.

그래서 Rhythm은 **에너지, 감정, 활동을 30초 안에 기록**하고, 그 데이터를 **Wave Graph**로 표현합니다. 사용자는 오늘 탭에서 에너지와 감정 키워드, 활동 태그를 선택하고, 저장 후 파동 그래프가 어떻게 바뀌는지 확인합니다. 이후 히스토리에서 날짜별 기록을 다시 보고, 패턴 탭에서 평균 에너지와 자주 나온 감정, 활동을 확인하는 흐름입니다.

현재는 중간 발표용 Flutter Web 데모로 입력 화면, 감정별 파동 그래프, 히스토리, 패턴 카드까지 구현했습니다.

기술 스택은 **Flutter, Riverpod, Isar, Layered Architecture**로 정했습니다.  
Flutter는 CustomPainter 기반 시각화를 만들기 좋고, 최종 구조에서는 Riverpod, Isar, Layered Architecture를 적용해 화면, 상태, 저장소 역할을 분리할 계획입니다. 특히 Wave Graph를 그리는 부분과 감정 데이터를 계산하는 부분을 분리하는 것을 핵심 원칙으로 잡았습니다.

사용자 시나리오는 오늘의 리듬을 남기고, 저장된 기록을 다시 확인한 뒤, 패턴을 통해 생활 리듬을 돌아보는 흐름으로 구성했습니다. GitHub Pages에서는 발표자료와 WBS/Gantt 페이지, Flutter Web 데모를 바로 확인할 수 있게 준비했습니다.

기말까지는 로컬 저장 구조를 안정화하고, 감정-활동 상관 분석과 주간 리듬 리포트를 추가해서 단순 기록 앱이 아니라 생활 리듬을 분석하는 앱으로 발전시키겠습니다. 추가 후보로는 월간 캘린더 뷰와 날씨 API 연동도 검토하고 있습니다.

감사합니다. 질문 받겠습니다.

## 1. 왜 Flutter를 선택했나요?

이 앱은 감정 데이터를 파동 그래프로 시각화하는 것이 핵심이라, 직접 UI를 그릴 수 있는 `CustomPainter`가 중요했습니다. Flutter는 모바일 앱과 웹 데모를 하나의 코드베이스로 만들 수 있고, 나중에 iOS와 Android 확장도 가능해서 선택했습니다. 이 결정은 `ADR-0001`에 기록했습니다.

## 2. 이 화면의 데이터는 어디서 오나요?

사용자가 오늘 탭에서 입력한 에너지 레벨, 감정 키워드, 활동 태그가 데이터의 출발점입니다. 현재 데모는 일부 샘플 데이터와 입력값을 함께 사용하고 있고, 최종 구조에서는 Isar 로컬 DB에 저장한 기록을 히스토리와 패턴 화면에서 불러오는 방식입니다.

## 3. 지금 가장 큰 문제는 무엇인가요?

가장 큰 위험은 Wave Graph가 앱의 차별점이지만 구현 난도가 있다는 점입니다. 단순히 예쁜 그래프가 아니라 감정 키워드에 따라 파동의 형태가 달라져야 해서, 성능과 설명 가능성을 같이 고려해야 합니다. 그래서 Must 기능부터 완성하고, 시각 효과는 제가 이해하고 수정할 수 있는 범위로 조절하고 있습니다.

## 4. 어디서부터 어디까지 AI가 만들었나요?

기획 문서 초안, 발표자료 정리, Flutter 데모 코드 일부는 AI 도움을 받았습니다. 하지만 그대로 제출한 것이 아니라, 제가 프로젝트 방향에 맞게 수정했고, WBS 일정, ADR 내용, 화면 구성, 감정 키워드별 파동 디자인 같은 부분은 직접 검토하면서 반영했습니다.

## 5. 본인이 직접 짠 부분은 어떤 거예요?

직접 정한 부분은 앱의 주제와 문제 정의, Must 기능 범위, 기술 선택 방향, WBS 일정, 그리고 데모 화면에서 어떤 흐름을 보여줄지입니다. 코드에서는 감정 입력 흐름, 파동 그래프가 어떤 데이터에 반응해야 하는지, 발표용 화면 구성과 디자인 수정 방향을 직접 잡았습니다.

## AI 활용 질문이 나왔을 때 핵심 문장

> AI를 사용한 것은 맞지만, 제가 설명하지 못하는 코드는 넣지 않으려고 했습니다. 그래서 이번 중간 발표에서는 완성도보다도, 제가 구조와 선택 이유를 이해하고 있는지를 보여주는 데 집중했습니다.

## 기술 스택 쉽게 설명하기

### Flutter

Flutter는 Google에서 만든 앱 개발 프레임워크다.  
하나의 코드베이스로 Android, iOS, Web 같은 여러 플랫폼에 앱을 만들 수 있다.

Rhythm에서 Flutter를 선택한 이유는 화면을 직접 그리는 기능이 좋기 때문이다.  
특히 `CustomPainter`를 사용하면 일반적인 버튼이나 리스트뿐만 아니라, 감정 데이터를 바탕으로 움직이는 Wave Graph 같은 시각 요소를 직접 만들 수 있다.

발표 때는 이렇게 말하면 된다.

> Flutter는 한 번 만든 코드로 웹 데모와 모바일 앱까지 확장할 수 있고, CustomPainter로 파동 그래프를 직접 그리기 좋아서 선택했습니다.

주의할 점:

- Flutter는 디자인을 빠르게 만들 수 있지만, 구조 없이 만들면 화면 코드가 커질 수 있다.
- 그래서 Riverpod과 Layered Architecture로 상태와 역할을 분리하려고 했다.

### Riverpod

Riverpod은 Flutter에서 상태 관리를 도와주는 라이브러리다.  
여기서 상태란 현재 선택된 에너지, 감정 키워드, 저장된 기록 목록처럼 화면에 영향을 주는 데이터를 말한다.

예를 들어 사용자가 감정 키워드를 선택하면 화면의 Wave Graph가 바뀌어야 한다.  
이때 화면 코드 안에서 모든 데이터를 직접 관리하면 복잡해지기 쉽다. Riverpod을 사용하면 화면은 상태를 읽고, 상태 변경 로직은 별도로 관리할 수 있다.

발표 때는 이렇게 말하면 된다.

> Riverpod은 화면과 상태 로직을 분리하기 위해 선택했습니다. 사용자가 입력한 감정이나 에너지 값이 바뀌면, 그 상태를 기준으로 화면이 다시 그려지도록 관리합니다.

주의할 점:

- Riverpod 자체가 앱 구조를 자동으로 좋게 만들어주지는 않는다.
- 어떤 상태를 어디에 둘지 직접 정해야 한다.
- 그래서 ViewModel이나 UseCase와 함께 쓰는 방향으로 설계했다.

### Isar

Isar는 Flutter에서 사용할 수 있는 로컬 데이터베이스다.  
서버 없이도 사용자의 휴대폰이나 브라우저 환경에 데이터를 저장할 수 있다.

Rhythm은 매일의 감정 기록을 저장하고 다시 불러와야 한다.  
하지만 중간 발표 단계에서는 백엔드 서버까지 만드는 것보다, 먼저 로컬에 기록을 안정적으로 저장하고 조회하는 것이 더 중요하다고 판단했다.

발표 때는 이렇게 말하면 된다.

> Isar는 로컬 DB라서 백엔드 없이도 사용자의 기록을 저장할 수 있습니다. Rhythm은 개인 감정 기록 앱이기 때문에, 우선 오프라인에서도 기록을 남길 수 있는 구조를 선택했습니다.

주의할 점:

- Isar를 쓰면 서버 로그인이나 여러 기기 동기화는 바로 해결되지 않는다.
- 대신 개인 프로젝트 범위에서는 로컬 저장과 빠른 조회에 집중할 수 있다.
- 나중에 필요하면 백엔드 연동을 추가하는 방향으로 확장할 수 있다.

### Layered Architecture

Layered Architecture는 앱을 역할별 계층으로 나누는 설계 방식이다.  
쉽게 말하면 화면, 상태 처리, 핵심 규칙, 데이터 저장을 한 파일에 섞지 않고 분리하는 방식이다.

Rhythm에서는 다음처럼 나누려고 한다.

| 계층 | 역할 | Rhythm 예시 |
|------|------|-------------|
| Presentation | 화면 표시 | 오늘 탭, 히스토리 탭, Wave Graph UI |
| Application | 화면 상태와 흐름 제어 | 기록 저장 버튼 처리, 선택된 감정 상태 관리 |
| Domain | 핵심 규칙 | 감정 키워드를 Wave Graph 설정으로 변환 |
| Data | 저장소 구현 | Isar에 기록 저장, 기록 목록 불러오기 |

이 구조를 선택한 이유는 Wave Graph가 앱의 핵심 기능이기 때문이다.  
그래프를 그리는 코드와 감정 데이터를 계산하는 코드를 분리해두면, 나중에 그래프 디자인이 바뀌어도 전체 앱 구조를 크게 고치지 않아도 된다.

발표 때는 이렇게 말하면 된다.

> Layered Architecture는 화면, 로직, 저장소 역할을 나누기 위해 선택했습니다. 특히 Wave Graph를 그리는 부분과 감정 데이터를 계산하는 부분을 분리해서, 나중에 기능을 바꿔도 유지보수하기 쉽게 만들고 싶었습니다.

주의할 점:

- 작은 앱에서는 구조가 조금 과해 보일 수 있다.
- 하지만 이번 프로젝트는 기획 문서, ADR, WBS까지 포함한 개인 프로젝트라서 구조를 설명 가능하게 만드는 것이 중요하다.
- 모든 계층을 완벽하게 구현하기보다, 핵심 흐름부터 계층을 나누는 방식으로 진행하면 된다.
